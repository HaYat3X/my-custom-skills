# 技術レビュー結果

## 総合評価
要修正

## 指摘事項

### 🔴 Critical（必須対応）

- **Shapefile クライアントサイド処理の実現性**
  ShapefileはZIP圧縮された複数ファイル（.shp / .dbf / .prj / .shx 等）の組み合わせで成立する。`shpjs` 等のライブラリでクライアントサイド処理は可能だが、大規模ファイルやマルチパートポリゴンで顕著なメモリ消費・フリーズが発生するリスクがある。利用可能ファイルサイズ上限と対応CRS（座標参照系）の種類を明示すること。プロジェクションファイル（.prj）の内容次第では座標変換ライブラリ（`proj4js`）が必須になり、変換精度の保証方針も決める必要がある。

- **GeoTIFF のクライアントサイドレンダリング上限が未定義**
  `geotiff.js` + `georaster-layer-for-leaflet` を使えばブラウザ内描画は可能だが、数十MB超の衛星画像やマルチバンドTIFFではブラウザメモリが数GBに達しクラッシュする。「GeoTIFF・ラスター」という記述のまま要件化するのはリスクが高く、対応するファイルサイズ上限・バンド数・ビット深度を仕様に明記すること。

- **URL/API データ取得とCORS問題の矛盾**
  F-002 の備考に「CORSエラー発生時はメッセージを表示」とあるが、外部APIのほとんどはCORSを許可していないため、実運用でほぼ機能しない。一方でスコープ外に「サーバーサイド処理なし」と明記されており矛盾する。US-003 は Must 要件であるため、プロキシサーバー（Cloudflare Workers・Lambda@Edge 等）導入の検討か、対応データソースを自組織管理のCORS許可済みエンドポイントに限定する旨の制約追記が必須。

- **パフォーマンス要件の測定条件が不明確**
  「GeoJSON 10MBを3秒以内に表示」は受け入れ基準として機能しない。以下を定義すること。
  - 計測端末スペック（CPU・メモリの基準値）
  - ネットワーク帯域（ローカルファイル選択前提か、URLダウンロード含むか）
  - フィーチャー数・ジオメトリ複雑度の条件
  - 「表示完了」の定義（初期フレーム描画か、全フィーチャー描画完了か）

- **大量フィーチャー時のレンダリング戦略が未定義**
  Leaflet.js はDOMベースのため数万フィーチャーで応答不能になる。MapLibre GL JSはWebGLで有利だがメモリ限界はある。`supercluster` によるクラスタリング・`flatgeobuf` による部分ロードなど、大量データ対応の基本方針を技術スタック選定と同時に確定すること。

### 🟡 Warning（推奨対応）

- **地図ライブラリ選定の先送りリスク**
  「Leaflet.js または MapLibre GL JS を推奨」と曖昧に記載されているが、GeoTIFF・コロプレス・大量フィーチャーの要件を総合すると MapLibre GL JS が優位。選定を実装フェーズに持ち越すと、両者で検証コストが二重にかかる。要件定義段階で選定基準と決定を確定させること。

- **CSV lat/lon 自動検出ロジックの定義不足**
  「lat/lon 列を自動検出」とあるが、列名バリエーション（latitude/longitude、y/x、緯度/経度、lat_wgs84 等）・数値と文字列の混在・フォーマット（度分秒 vs. 十進度）の扱いが未定義。自動検出失敗時にユーザーが手動でマッピングできる UI が必要か否かも明記すること。

- **WMS/WFS 対応の工数過小評価リスク**
  F-002 に「WMS/WFSエンドポイント」とあるが、両者はプロトコルが全く異なり実装工数が大幅に異なる。WFS はGeoJSON相当の取得が可能だが WMS はタイル/画像取得であり別種の実装が必要。Must 要件（US-003）のスコープに含まれるか明確化し、対応バージョン（WMS 1.1/1.3、WFS 2.0等）も指定すること。

- **共有URL（F-012）の URL 長制限問題**
  複数レイヤー・スタイル設定・フィルタ条件・ビューポート状態を全てシリアライズすると URL が数KB を超え、ブラウザの URL 長制限（概ね 2,000〜8,000文字）に抵触する。Nice 要件ながら実装工数が大きく、Base64 圧縮・SessionStorage 等の設計が必要。

- **OSM タイルサーバーの利用ポリシーリスク**
  OpenStreetMap 標準タイルサーバー（tile.openstreetmap.org）は高トラフィックの商用利用を禁止しており、大量アクセス時にブロックされる。組織内展開・商用利用を想定する場合は MapTiler Cloud・自前 tile server（PMTiles + Cloudflare R2 等）への切り替えを計画すること。

- **ブラウザ対応の Safari 除外**
  Chrome / Firefox / Edge のみが対象で Safari が除外されている。macOS・iOS 利用者が業務担当者に含まれる場合、WebGL の挙動差異や `cross-origin isolation` 要件で問題が生じる可能性がある。意図的な除外であれば要件書に明記し、関係者合意を得ること。

### 🟢 Info（参考情報・提案）

- **クライアントサイド完結の設計思想は高く評価できる**
  データをサーバーに送信しない方針はプライバシー・データガバナンスの観点で優れた選択。ただし将来的にデータ永続化（保存・チーム共有）ニーズが生じた際、この制約が大規模な設計変更を要求する点を事前にステークホルダーへ共有しておくことを推奨。

- **フィルタリング UI の複雑度に注意**
  「属性・演算子・値でフィルタ条件を設定」はクエリビルダー UI の実装を意味し、数値・文字列・日付型の演算子差異、複合条件（AND/OR）の対応範囲によって工数が大きく変動する。MVP では単一条件に限定し、複合条件は別フェーズとすることを推奨。

- **Streaming GeoJSON の検討**
  10MB 超のデータに対して Fetch API の ReadableStream を活用したストリーミングパース（`geojson-stream` 等）を採用すると、表示開始までの体感時間を短縮できる。パフォーマンス目標の達成手段として設計フェーズで検討を推奨。

- **アクセシビリティ要件の具体化**
  「キーボードナビゲーションを考慮」は曖昧。地図操作（パン・ズーム）のキーボード対応範囲、ARIA ランドマーク対応、WCAG AA のコントラスト比準拠など、対応レベルを明記することで実装範囲が明確になる。

## 総評

クライアントサイド完結の設計思想は一貫しており、MVPのスコープ切り分けも概ね妥当。しかしGeoTIFFのメモリ上限・ShapefileのCRS変換・URL取得時のCORSという3点はCriticalな実現性リスクであり、技術選定と同時に解決策を確定させなければ主要機能がデモ止まりになる危険がある。パフォーマンス要件の測定条件も早急に定量化が必要。
