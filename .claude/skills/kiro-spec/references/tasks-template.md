# tasks.md テンプレート

以下のフォーマットで `specs/tasks.md` を生成する。
**必ず `specs/design.md` を読み込んでから生成すること。**

---

## テンプレート本文

```markdown
# タスク一覧: {機能名 / プロジェクト名}

作成日: {YYYY-MM-DD}
参照: [requirements.md](./requirements.md) / [design.md](./design.md)

## 進捗サマリー

- 総タスク数: {N}
- 完了: 0 / {N}
- Must: {N}件 / Should: {N}件 / Could: {N}件

---

## タスク一覧

### Phase 1: 基盤整備

- [ ] **T-01** 〔Must〕{タスク名}
  - 内容: {何をするか具体的に}
  - 成果物: `{ファイルパス or 機能名}`
  - 見積: {30min / 1h / 2h / 半日}
  - 依存: なし

- [ ] **T-02** 〔Must〕{タスク名}
  - 内容: {何をするか}
  - 成果物: `{ファイルパス}`
  - 見積: {時間}
  - 依存: T-01完了後

### Phase 2: コア機能実装

- [ ] **T-03** 〔Must〕{タスク名}
  - 内容: {何をするか}
  - 成果物: `{ファイルパス}`
  - 見積: {時間}
  - 依存: T-02完了後

- [ ] **T-04** 〔Should〕{タスク名}
  - 内容: {何をするか}
  - 成果物: `{ファイルパス}`
  - 見積: {時間}
  - 依存: T-03完了後

### Phase 3: UI / 統合

- [ ] **T-05** 〔Must〕{タスク名}
  - 内容: {何をするか}
  - 成果物: `{ファイルパス}`
  - 見積: {時間}
  - 依存: T-03, T-04完了後

### Phase 4: テスト・仕上げ

- [ ] **T-06** 〔Should〕動作確認・バグ修正
  - 内容: 全機能の動作確認、エラーハンドリング確認
  - 成果物: 修正済みコード
  - 見積: 1h
  - 依存: Phase 3完了後

- [ ] **T-07** 〔Could〕{追加改善タスク}
  - 内容: {何をするか}
  - 成果物: {成果物}
  - 見積: {時間}
  - 依存: T-06完了後

---

## 優先度の定義

| 優先度 | 意味 |
|--------|------|
| Must | リリース必須。これがないと機能しない |
| Should | 重要だが一時的に省略可能 |
| Could | あると嬉しいが後回しでよい |

## タスク更新ルール

実装完了時は `[ ]` → `[x]` に変更する。
途中で仕様変更があった場合は tasks.md を更新してから実装を続ける。
```

---

## 生成時の注意点

- 1タスク = 30分〜半日で完結するサイズに分割する
- 「バックエンド実装」のような大きな塊はさらに分割する
- 依存関係は明示的に書く（並行作業可能なものも明示する）
- Must タスクだけで最小動作する構成を意識する
- 合計見積もり時間をサマリーに記載すると全体感が掴みやすい
